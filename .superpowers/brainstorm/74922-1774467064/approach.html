WebGL Hero — Technical Approach

The hero animates: particle field → sacred geometry → Phönix sphere → text reveal (scroll-driven). Which implementation path fits best?

A

Pure Three.js

Raw WebGL. Shader-level particle morphing, full GPU control. Most faithful to the vision — real 3D geometry, true volumetric particles. Heaviest to build and maintain.

Pros

  • Maximum visual fidelity
  • True 3D depth & lighting
  • Shader-driven morphing

Cons

  • Lives outside React lifecycle
  • Hardest to maintain
  • Longest build time
B

React Three Fiber (r3f)

Three.js inside React. Fits the existing Vite + React stack naturally. Component-based scene graph, hooks for animation, GSAP-compatible scroll triggers. Recommended.

Pros

  • Fits existing stack
  • Component-based, maintainable
  • Same visual capability as pure Three.js

Cons

  • Adds r3f + drei dependencies
  • Slightly more abstraction overhead
C

Canvas 2D + GSAP

No WebGL. Particles and geometry drawn on a 2D canvas, animated with GSAP (already in the project). Simpler, more accessible, zero new dependencies.

Pros

  • No new dependencies
  • Works everywhere, including low-end mobile
  • Simpler codebase

Cons

  • No real 3D depth
  • Less immersive particle morphing
  • Can't do true volumetric effects
