Three Approaches to the WebGL Hero

React Three Fiber · Idle drift → scroll morphs · Mobile SVG fallback · Implied sphere

A

Single Unified Shader

One particle system, one GLSL shader that morphs between 4 states (scattered → Fibonacci → Flower of Life → sphere cluster) using attribute lerping. Pure GPU-driven. Smoothest transitions, hardest to build.

Pros

  • Perfectly continuous morph
  • Best performance (all on GPU)
  • No geometry swaps mid-sequence

Cons

  • Requires custom GLSL shaders
  • Hard to debug, long build
  • Overkill for this use case
B

GSAP-Scrubbed Position Targets ⭐ Recommended

Particles are Three.js Points. Each particle has a "home" position for each stage (random → fibonacci → flower-of-life → cluster). GSAP ScrollTrigger scrubs between position targets. No custom shaders — just lerping BufferGeometry attributes with GSAP timelines.

Pros

  • Uses GSAP already in project
  • No custom shaders needed
  • Each stage independently tunable
  • Scroll scrub is native GSAP

Cons

  • CPU-side position updates each frame
  • Morph slightly less fluid than shader
C

Stage-by-Stage Opacity Swap

Three separate particle systems, one per key stage (field, geometry, sphere). Cross-fade between them on scroll rather than morphing positions. Simpler logic, but transitions are fades not true morphs.

Pros

  • Simplest to build
  • Easiest to tweak each stage independently

Cons

  • Cross-fades, not true particle morphing
  • Loses the "chaos becoming order" visual
  • Betrays the core concept
